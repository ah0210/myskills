# Code Review 标准化评分标准 v1.0

> 所有代码评审必须严格按此标准逐项打分，不能只说"好"或"不好"

---

## 🎯 评审指令

你是高级技术专家。请严格按照以下 5 个维度对代码进行评审，每项 0-10 分，低于 6 分的项必须给出具体的可执行的修改建议。

---

## 📋 5 维评审标准

### 1. 🐛 正确性（权重 ×3，最重要！）

| 分数 | 标准 |
|------|------|
| 10 | 没有任何潜在问题，所有边界情况都考虑到了 |
| 9 | 极端边界情况可以优化，不影响正确性 |
| 8 | 业务逻辑正确，边缘情况处理不完整 |
| 6-7 | 有小问题，不影响主流程 |
| <6 | ❌ 有明确的 bug，必须修改 |

**必须检查：**
- 空指针 / null / undefined
- 数组越界
- 整数溢出
- 竞态条件 / 并发问题
- 异常处理
- 业务逻辑正确性

---

### 2. ⚡ 性能（权重 ×2）

| 分数 | 标准 |
|------|------|
| 10 | 达到理论最优时间/空间复杂度 |
| 8-9 | 复杂度合格，常数可以优化 |
| 6-7 | 复杂度偏高，但在可接受范围 |
| <6 | ❌ 有 O(n²) 之类的明显可优化点 |

**必须检查：**
- 时间复杂度
- 空间复杂度
- 不必要的循环/重复计算
- 内存泄漏风险
- 数据库 N+1 查询

---

### 3. 📖 可读性（权重 ×2）

| 分数 | 标准 |
|------|------|
| 10 | 像散文一样好懂，不需要注释 |
| 8-9 | 命名准确，结构清晰 |
| 6-7 | 能读懂，但需要费点劲 |
| <6 | ❌ 写得像加密的，根本看不懂 |

**必须检查：**
- 变量/函数命名
- 注释质量（不要"1+1=2"这种废话注释）
- 函数长度（不超过一屏）
- 代码结构分层
- 符合团队编码规范

---

### 4. 🔒 安全性（权重 ×2）

| 分数 | 标准 |
|------|------|
| 10 | 按 OWASP 标准审计过，无风险 |
| 8-9 | 常规安全问题都考虑到了 |
| 6-7 | 有小风险，但不是高危 |
| <6 | ❌ 有 SQL 注入/XSS/CSRF 等明显漏洞 |

**必须检查：**
- 输入验证
- SQL 注入
- XSS / CSRF
- 敏感信息泄露
- 权限控制

---

### 5. 🧪 可测试性（权重 ×1）

| 分数 | 标准 |
|------|------|
| 10 | 每个函数都可以独立写单元测试 |
| 8-9 | 主要逻辑可测，少量耦合 |
| 6-7 | 可以测，但需要 mock 很多东西 |
| <6 | ❌ 一坨面条代码，根本没法写测试 |

---

## ⚠️ 评审铁则

1. **低于 6 分必须给具体修改代码，不能只说问题**
2. **不能说空话**：不许说"代码写得不错"、"还可以"这种废话
3. **对事不对代码**：只说具体问题，不说抽象感受
4. **可验证**：所有问题必须是可复现、可验证的
